## 为什么

当前操作员标注体验需要重新设计，确保操作员可以稳定找到本人负责的计划、阅读病例、对比 A/B 输出，并提交可追溯的标注结论。本变更用于在进入前后端开发前，先明确操作员标注模块的产品边界、能力规范与实施顺序。

## 范围

### 范围内

- 从零重建操作员侧计划入口与标注流程。
- 定义仅操作员可访问的路由、页面、数据契约、权限边界、加载态、空状态、校验态、完成态与异常态。
- 按前端优先的方式实施，先使用 mock 数据和本地状态完成 Pencil 还原与体验验收，再进入后端开发。
- 实现操作员计划发现、下一条任务获取、标注提交所需的后端 API、schema、service、repository 与持久化逻辑。
- 继续沿用项目级认证、角色授权、计划负责人、病例数据、质控规则、A/B 展示映射、标注结果语义等全局领域约束。
- 在本地前后端联调通过后，补充 Docker Compose 部署验证任务。

### 范围外

- 复用或增量修补现有操作员标注模块实现。
- 管理员计划创建、CSV 导入、统计报表、成员管理、规则管理行为；但操作员 API 可读取这些模块已产生的数据。
- 新认证方式、用户注册、密码找回、单点登录、审计日志、多租户隔离或复杂 RBAC。
- 导出、打印、批量标注、快捷键模式、协作标注、操作员侧历史结果编辑。
- 模型推理、AI 评分、自动规则生成或训练流程。

## 变更内容

- **BREAKING**：操作员模块按全新实现处理；现有操作员 UI 与任务接口代码只能作为项目结构参考，MUST NOT 作为新设计的行为依据。
- 新增操作员计划入口页，用于展示本人负责计划、进度信息，并进入可执行的标注工作。
- 新增操作员标注工作台，包含病历内容、展示后的 A/B 输出、质控规则上下文、结论选择、原因选择、其他原因校验、备注和提交反馈。
- 新增确定性的任务队列行为，用于获取操作员本人负责的活跃计划中的下一条未标注病例。
- 新增标注提交行为，保证同一病例同一操作员只能成功提交一次，并在提交后进入下一条任务或完成态。
- 新增前端 mock 数据流程，在后端接入前完成页面状态与交互验收。
- 新增后端 FastAPI 路由与持久化契约，覆盖操作员计划列表、任务获取和标注提交。

## 能力

### 新增能力

- `operator-plan-entry`：操作员负责计划发现、进度展示、进入标注工作流。
- `operator-task-queue`：按权限和计划状态获取下一条可标注任务，并处理无任务完成态。
- `operator-annotation-workbench`：三栏标注工作台展示、内容渲染、质控规则上下文与页面状态。
- `operator-annotation-submission`：标注表单校验、单次提交持久化与提交后推进。

### 修改能力

- 无。认证、管理员计划/导入/报表/成员/规则等既有全局能力继续保持权威；本变更新增重新设计后的操作员模块契约。

## 影响

- 前端：`/operator` 下的 React Router 路由、新操作员页面与组件结构、操作员 API client、操作员类型、mock fixtures 与测试。
- 设计：`design/pages/operator.pen` 定义必须覆盖的页面状态与还原目标。
- 后端：`/api/operator/`* FastAPI 路由边界、Pydantic schemas、计划分配/任务队列/标注写入相关 service 与 repository。
- 数据库：使用既有 `plans`、`case_records`、`annotations`、`quality_rules`、`users` 表；仅在实现证明必要时补充索引或字段。
- 验证：前端 mock 验收先于后端开发；后端本地验证使用 conda `aicompare` 环境，不使用 `venv`。
- 部署：Docker Compose 构建与 smoke test 必须覆盖重建后的操作员流程。

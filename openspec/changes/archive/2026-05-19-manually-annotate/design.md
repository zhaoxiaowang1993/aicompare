## Context

当前系统以“对比模式”为唯一业务形态：计划导入四列 CSV，`CaseRecord` 必须包含病历原文、智能体 A/B 输出和固化后的 A/B 展示映射；操作员工作台固定为三栏对比；提交结果写入 `Annotation`，每个病例每个操作员最多一条对比结论；管理员统计围绕 A/B 结论和原因分布展开。

手动标注模式改变的是业务形态，而不是单个页面字段。它仍共享计划、病例、负责人、计划关闭、质控规则和进度概念，但病例输入不包含智能体输出，标注结果从“一条对比结论”变成“一个病例完成记录 + 零到多条问题条目”。其中“无问题”也是标注员主动提交的完成状态，必须计入完成进度。

受影响设计稿路径：

- `design/pages/admin-plan-list.pen`：计划列表展示标注类型。
- `design/pages/admin-plan-detail.pen`：计划详情、统计概览、标注明细按模式展示。
- `design/pages/operator.pen`：操作员计划列表和手动标注工作台。

## Goals / Non-Goals

**Goals:**

- 在计划层引入 `annotation_type`，支持 `comparison` 和 `manual` 两种模式。
- 保持现有对比模式的导入、工作台、提交、统计行为兼容。
- 为手动模式提供两列 CSV 导入、两栏工作台、多条问题条目提交和“无问题”主动完成。
- 将完成进度定义在病例级别，避免手动模式因多条问题条目导致进度统计失真。
- 管理员报表按计划模式分流：对比模式保留决策/原因分布，手动模式首期仅提供完成情况概览和手动明细，不提供问题类型完成情况。

**Non-Goals:**

- 不引入多人协同、自动保存、标注版本历史或审核流。
- 不允许管理员代替标注员完成手动标注。
- 不在本次实现规则版本冻结；手动明细通过提交时快照保证历史可读。
- 不在首期提供手动标注明细导出、打印或复杂统计口径。
- 不把 `openspec/specs/SPEC.md` 作为详细需求承载；它只在 specs/design 稳定后更新项目级概览。

## Decisions

### 1. 以计划类型作为顶层分流

新增 `Plan.annotation_type`，取值为 `comparison` 或 `manual`。已有计划和未显式传入类型的新计划默认视为 `comparison`。

选择该方案的原因：

- 计划是导入模板、任务 payload、工作台、提交校验和报表口径的共同边界。
- 按计划分流可以防止同一计划中混入两类病例和两类标注结果。
- 默认 `comparison` 能降低对现有数据和测试的破坏。

备选方案：

- 在病例或标注记录上放类型字段：会让同一计划内出现混合语义，进度和报表更难保证一致。
- 独立创建一套手动计划表：隔离更强，但会复制计划生命周期、负责人和权限逻辑。

### 2. 复用 `CaseRecord` 存储病例基础数据

`CaseRecord` 继续作为计划内病例的统一表。`hospitalization_no`、`record_text`、`import_batch_id` 仍为通用字段；`agent_a_output`、`agent_b_output`、`display_a_source`、`display_b_source` 改为仅对 `comparison` 计划必填，对 `manual` 计划为空。

选择该方案的原因：

- 计划总病例数、去重、队列顺序和病例权限可以继续复用。
- 手动模式与对比模式都以“住院号 + 病历原文”为任务基础。
- 避免为手动模式复制一套 case repository 和进度查询基础设施。

备选方案：

- 新建 `ManualCaseRecord`：能避免 nullable 字段，但会让计划统计、导入摘要和任务队列出现两套实现。
- 将手动病例塞入现有 A/B 输出字段为空字符串：绕过数据库约束但语义不清，容易让对比工作台误渲染空输出。

### 3. 手动标注采用“病例完成记录 + 条目表”

新增病例级完成记录表，例如 `ManualCaseAnnotation`：

- `plan_id`
- `case_id`
- `operator_user_id`
- `result`：`has_issues` 或 `no_issue`
- `created_at` / `updated_at`
- 唯一约束：`(case_id, operator_user_id)`

新增条目表，例如 `ManualAnnotationEntry`：

- `manual_annotation_id`
- `plan_id`、`case_id`、`operator_user_id` 的冗余索引用字段
- `source_text`：标注员划选的病历原文片段
- `start_offset`：原文片段在清洗后 `record_text` 中的起始字符位置，包含该位置
- `end_offset`：原文片段在清洗后 `record_text` 中的结束字符位置，不包含该位置
- `quality_rule_id`
- `quality_rule_category_snapshot`
- `quality_rule_content_snapshot`
- `quality_rule_score_snapshot`
- `suggestion`
- `notes`
- `created_at`

提交规则：

- `result=has_issues` 时必须至少包含一条 entry。
- `result=no_issue` 时 entries 必须为空，且该病例视为完成。
- 每条 entry 的 `start_offset` / `end_offset` 必须落在当前病例清洗后 `record_text` 的有效范围内，并且 `source_text` 应与该区间对应文本在换行规范化后保持一致。
- 同一病例同一操作员重复提交返回冲突语义。

选择该方案的原因：

- “无问题”没有问题条目，必须有病例级完成记录才能计入进度。
- 多条问题条目属于同一次病例完成结果，适合挂在同一个完成记录下。
- 规则快照可以避免质控规则后续编辑或软删除后，历史明细失去上下文。

备选方案：

- 复用现有 `Annotation` 表：字段含义会被迫 nullable 或重载，A/B 决策枚举与手动条目不匹配。
- 只建 entry 表：无法表达“无问题”完成，也无法稳定去重。
- 把条目 JSON 存在完成记录中：写入简单，但分页、筛选、明细展示和未来统计扩展会变差。

### 4. CSV 导入采用模式化 parser

导入入口保留现有路由形态，但创建计划时增加 `annotation_type` 表单字段；向已有计划追加导入时根据目标计划的 `annotation_type` 选择模板。

- `comparison` 模板保持：`住院号`、`病历内容`、`智能体A输出`、`智能体B输出`。
- `manual` 模板为：`住院号`、`病历内容`。
- 清洗、行级错误、同计划住院号去重、导入摘要继续复用。
- 手动导入不会生成 A/B display mapping。

选择该方案的原因：

- 对管理员来说仍是“创建计划并上传 CSV”的同一流程。
- 模板差异由计划类型决定，前后端规则一致。
- 不改变对比模式现有 CSV 合同。

备选方案：

- 为手动模式新增完全独立上传接口：隔离明确，但前端和后端重复较多。
- 通过自动识别 CSV 列数判断模式：容易出现误判，也绕过管理员显式选择。

### 5. 任务与提交 API 使用模式分流合同

操作员计划列表和计划详情 payload 增加 `annotation_type`。下一条任务接口继续以计划为入口，但响应应带有模式判别字段，使前端根据 `annotation_type` 渲染对应工作台。

建议合同：

- `GET /api/operator/plans/{plan_id}/tasks/next`
  - `comparison`：返回现有病历、输出 A/B、展示映射和质控规则。
  - `manual`：返回病例 id、住院号、病历原文和质控规则，不返回输出 A/B。
- `POST /api/operator/tasks/{case_id}/annotate`
  - 保留为对比模式提交入口，避免破坏既有客户端。
- `POST /api/operator/tasks/{case_id}/manual-annotate`
  - 新增手动模式提交入口，提交 `result` 和 entries。

选择该方案的原因：

- 任务读取可以共用队列入口，因为“下一条病例”的权限和关闭计划规则一致。
- 提交入口分开可以让 Pydantic schema 明确，不需要在同一路由内做复杂 union 校验。
- 现有对比提交接口不需要承担手动条目语义。

备选方案：

- 单一提交接口使用 discriminated union：路由更少，但错误语义和客户端类型会更复杂。
- 完全独立手动任务 API：隔离更彻底，但会复制权限、完成态和队列顺序逻辑。

### 6. 管理员报表按计划类型返回模式化结构

`GET /api/admin/plans/{plan_id}/stats` 根据计划类型返回对应统计：

- `comparison`：保留总量、已标注、待标注、完成率、决策分布、原因分布。
- `manual`：首期仅返回总量、已完成、待完成、完成率。首期不返回问题类型完成情况、规则分布或其他复杂看板数据。

`GET /api/admin/plans/{plan_id}/annotations` 根据计划类型返回对应列表明细：

- `comparison`：保留现有 A/B 输出、结论、原因结构。
- `manual`：返回手动病例完成记录的病例级聚合列表。一份住院病历即使包含多条手动问题条目，在计划详情页标注明细列表中也只展示一行。列表字段包含住院号、操作员、完成状态/结果、问题数、病例完整提交时间和查看详情入口，不展示原文片段、质控规则、修改建议、备注，也不展示 `start_offset` / `end_offset`。
- `manual` 列表的标注时间使用该病例完整提交的时间，即 `ManualCaseAnnotation` 的提交/创建时间；问题条目自身的创建时间只在详情页列表视图中展示。
- `no_issue` 完成记录仍返回一行病例级完成明细，显式标记 `result=no_issue`，问题数为 0。

手动模式的单病例详情使用独立详情数据，建议新增或等价提供 `GET /api/admin/plans/{plan_id}/annotations/{manual_case_annotation_id}`：

- 返回住院号、操作员、病例原文、病例完成状态、病例提交时间和该病例下全部手动问题条目。
- 每条问题条目包含 `source_text`、`start_offset`、`end_offset`、质控规则快照、修改建议、备注和条目创建时间。
- 管理员详情默认展示“工作台视图”：复用手动工作台的两栏结构，但为只读模式。左侧展示病历原文并根据 offset 黄色高亮问题原文；右侧展示只读标注卡片，并与高亮原文尽量保持同一水平位置。若高亮位置过近，继续使用非重叠平铺算法，保持原文位置靠前的卡片排在上方。
- 管理员详情支持切换到“列表视图”：表格列为病历原文、质控规则、修改建议、标注时间；其中标注时间为创建该问题条目卡片的时间。首期不提供详情内搜索筛选。设计稿必须复用 `design/components` 中的 Table / Pagination 组件样式，标注时间列使用正文文本样式。
- `no_issue` 详情中工作台视图展示病历原文但无高亮和问题卡片；列表视图展示空态。

选择该方案的原因：

- 管理员仍从计划详情进入统计和明细，不需要理解多个 API 入口。
- 前端可以根据 `annotation_type` 渲染不同表格和筛选器。
- 病例级聚合列表与进度口径一致，避免一个病例多条问题在计划详情列表里被误读为多份完成。
- “无问题”完成不会从明细中消失，同时有问题病例的多条规则细节仍可从详情页查看。
- 管理员只读工作台视图复用 offset 定位能力，便于审阅高亮原文与标注卡片的对应关系。

备选方案：

- 为手动模式新增独立报表路由：更显式，但计划详情页 API 编排更多。
- 让手动模式复用对比分布字段为空对象：短期简单，但语义容易误导。
- 在计划详情标注明细列表中直接按 entry 展示：能少一次详情跳转，但会让一个病例多行展示，破坏完成进度和列表筛选口径。

### 7. 前端以工作台组件分流，而不是在现有三栏组件内堆条件

前端保留现有对比工作台，并抽象命名为 comparison workbench；新增 manual workbench。

手动工作台布局：

- 顶部栏继续展示返回、计划名、状态、当前任务序号和进度。
- 左侧为病历原文独立滚动区域，title 区展示住院号和“查看质控规则”入口。
- 右侧为标注条目面板，包含条目列表、新建/编辑/删除入口、提交按钮。
- 左右两栏采用同一纵向画布语义，左侧病历原文滚动与右侧标注条目面板滚动保持同步；右侧卡片位置使用同一套 `start_offset` / `end_offset` 纵向坐标计算。
- 标注员在左侧选中文本后，系统在划线结束位置旁边展示“标注”浮层按钮；标注员取消划选或重新点击原文空白处时，该按钮同步消失。点击“标注”后，右侧条目自动带入只读的 `source_text`，新建表单在无视觉遮挡时与所选原文片段上边沿对齐，右侧面板自动滚动到该新建表单所在位置，并由系统记录隐藏的 `start_offset` 和 `end_offset`。
- 条目表单项只包含原文片段（只读）、质控规则单选、修改建议必填、备注选填；质控规则选择按规则类型分组展示，并支持按规则内容模糊搜索；`start_offset` 和 `end_offset` 不展示数值，也不允许标注员编辑。
- 在当前病例提交前，标注员可以对已创建的标注条目卡片进行查看、新增、编辑和删除；编辑时，表单在原条目卡片位置替换展示，不与原卡片同时显示；编辑保存成功后展示低干扰 toast 反馈。编辑和删除图标必须具备 hover / click 状态。
- 每个标注条目对应的病历原文片段在左侧默认以黄色高亮展示；右侧条目卡片上边沿在无视觉遮挡时与对应高亮原文片段上边沿保持同一水平位置，便于快速定位。
- 标注员点击某个右侧条目卡片时，左侧病历原文自动滚动到对应片段附近用于查看上下文，该条目对应的左侧原文片段切换为浅红色高亮；未选中的原文片段保持黄色高亮。条目卡片默认态为灰色边框，hover 态为蓝色边框，选中态为稳定的蓝色边框和浅蓝底色；鼠标移出后仍保持选中视觉，点击其他空白区域后恢复默认态。
- 当多个高亮原文片段距离过近导致右侧卡片可能重叠时，右侧卡片必须改为非重叠平铺展示，并保持“引用原文位置更靠前的卡片排在上方”的顺序。
- 右侧始终使用统一的“提交”按钮，不提供独立的“无问题，提交完成”按钮。
- 标注员点击“提交”时，无论当前是否存在标注条目，都先弹出二次确认。
- 如果标注条目数大于 0，确认文案为：`当前病历已标记{N}条质控问题，是否确认提交？提交后将不能新增/修改质控建议`。用户确认后提交 `has_issues` 结果和全部条目。
- 如果标注条目数为 0，确认文案为：`当前病历未标记质控问题，是否确认提交？提交后将不能新增/修改质控建议`。用户确认后提交 `no_issue` 结果并进入下一条任务。
- 当前病例提交成功后，该病例的质控条目不再允许新增、编辑或删除；后端通过重复提交冲突保证不可重提。

选择该方案的原因：

- 两类工作台的信息架构差异大，拆组件比条件渲染更可维护。
- 手动模式右侧是可编辑列表，不是输出对比栏。
- 条目卡片与原文高亮对齐可以降低标注员在长病历中来回定位的成本。
- 后续手动模式扩展统计口径或字段时，不会影响对比工作台。

备选方案：

- 在现有 `AnnotationWorkbench` 里按模式分支：文件会快速膨胀，交互状态容易互相污染。

### 8. 文本划选首期服务端持久化原文片段、换行标记和定位信息

首期提交持久化 `source_text`、`start_offset` 和 `end_offset`。`source_text` 应保留标注员选中文本中的换行信息，换行可统一规范为 `\n` 存储，以便管理员详情列表视图能够还原跨行原文片段。`start_offset` 和 `end_offset` 基于导入清洗后的同一份 `record_text` 计算，用于跨会话稳定恢复黄色高亮、条目卡片纵向对齐和防重叠布局；它们是系统定位元数据，不作为条目表单字段、管理员计划详情标注明细列表字段或管理员详情列表视图字段展示。

选择该方案的原因：

- 未来需要支持已提交病例再次编辑或管理员只读查看时，仅靠 `source_text` 文本匹配容易在重复片段中定位错误。
- `source_text` 用于审计可读，`start_offset` / `end_offset` 用于稳定定位和高亮，两者职责不同。
- offset 基于清洗后的 `record_text` 计算，可以避免导入前后换行和空白差异造成定位偏移。
- 跨行片段在质控标注中常见，保留换行信息能显著提高历史明细的可读性。

备选方案：

- 只存储 `source_text` 并在查看时做文本匹配：实现较轻，但重复文本、模板化病历内容和换行差异会导致高亮位置不稳定。

## Risks / Trade-offs

- [风险] 两种模式共享计划和病例表后，查询中可能遗漏 `annotation_type` 分流。→ 缓解：新增服务层 helper，所有统计、队列、提交从 plan type 分发；测试覆盖两种模式。
- [风险] 手动模式用 entry 数量计算进度会导致一个病例多条问题被计为多份完成。→ 缓解：进度只统计 `ManualCaseAnnotation` 病例级完成记录。
- [风险] “无问题”没有条目，管理员明细可能看不到这类完成。→ 缓解：手动计划详情标注明细列表按病例完成记录聚合展示，`no_issue` 也显式返回一行完成记录。
- [风险] 管理员计划详情标注明细若按问题条目展开，会让一个病例多行展示并影响完成口径理解。→ 缓解：列表仅展示病例级聚合行，问题条目进入单病例详情后再展示。
- [风险] 质控规则后续被编辑或删除后，历史手动条目引用内容变化。→ 缓解：提交时保存规则分类、内容、分值快照。
- [风险] 放宽 `CaseRecord` A/B 字段 nullable 可能掩盖对比模式导入缺字段。→ 缓解：数据库允许为空，业务层对 `comparison` CSV 和入库强校验必填。
- [风险] SQLite 既有表结构不易直接修改 NOT NULL 字段。→ 缓解：提供一次性迁移脚本或表重建迁移，并在本地/部署环境验证旧数据默认 `comparison`。
- [风险] 手动工作台文本选择和卡片对齐在复杂富文本渲染下体验不稳定。→ 缓解：服务端持久化基于清洗后 `record_text` 的 `start_offset` / `end_offset`，UI 使用 offset 恢复高亮并使用非重叠布局算法降级展示。

## Migration Plan

1. 数据结构迁移：
   - 为 `plans` 增加 `annotation_type`，既有记录填充 `comparison`。
   - 调整 `case_records` 的 A/B 输出和展示映射字段，使其可为空。
   - 新增 `manual_case_annotations` 和 `manual_annotation_entries`。
2. 后端兼容发布：
   - 先保持现有对比接口行为不变。
   - 增加按计划类型分流的 import、task、progress、report helper。
   - 增加手动提交接口和 schema。
3. 前端分阶段接入：
   - 先在计划创建、计划列表、计划详情和操作员计划列表展示类型。
   - 再接入手动工作台和手动提交。
   - 最后接入手动统计、病例级明细列表和管理员只读详情双视图。
4. 设计稿同步：
   - 更新 `design/pages/admin-plan-list.pen`、`design/pages/admin-plan-detail.pen`、`design/pages/operator.pen`。
   - 本次需求澄清后，需再次调整 `design/pages/admin-plan-detail.pen`：手动明细列表改为病例级聚合行，并补充管理员查看详情的只读工作台视图和列表视图。
   - 设计验收修复后，管理员计划列表不再展示草稿计划示例；手动完成概览仅保留完成情况卡片；操作员手动工作台补充默认态、新建卡片、编辑卡片、删除确认和组件化提交确认弹窗，且不提供右上角“新增条目”入口。
5. 回滚策略：
   - 代码可回滚到仅对比模式，但已经创建的手动计划和手动标注数据无法被旧代码完整使用。
   - 当前生产环境未上线，不要求备份 SQLite 数据库；如需回滚业务功能，应暂停创建手动计划并保留新表数据。

## Open Questions

无。当前已确认的约束如下：

- 手动模式管理员明细首期仅支持按操作员和标注日期过滤。
- 手动模式管理员计划详情页的标注明细列表按病例完成记录聚合展示，标注日期按病例完整提交时间过滤。
- 手动模式管理员查看详情首期提供只读工作台视图和列表视图；详情列表视图首期不提供搜索筛选。
- 手动模式管理员完成概览首期仅展示完成情况，不展示问题类型完成情况。
- 手动模式提交后，本期不支持编辑和重提。
- `no_issue` 表示此病例文书没有触发任何质控规则，不需要备注字段。
- 手动模式与对比模式共用质控规则，本期沿用对比模式相同的规则展示逻辑。
